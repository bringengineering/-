# 브링엔지니어링 — 보안 가이드

**최종 수정일:** 2026-03-10

---

## 보안 원칙

1. **최소 권한**: 업무에 필요한 최소한의 접근 권한만 부여
2. **암호화**: 중요 데이터는 저장/전송 시 암호화
3. **기록**: 누가 언제 무엇에 접근했는지 기록
4. **교육**: 팀원 전원이 보안 기본 규칙 숙지

---

## 계정 & 접근 관리

### 비밀번호 규칙
- 최소 12자, 대소문자+숫자+특수문자 포함
- 서비스마다 다른 비밀번호 사용
- **비밀번호 관리자 사용 필수** (1Password, Bitwarden 등)
- 3개월마다 주요 비밀번호 변경

### 2단계 인증 (2FA)
다음 서비스는 **반드시** 2FA 활성화:
- [ ] GitHub
- [ ] 클라우드 (AWS/GCP)
- [ ] 이메일
- [ ] 슬랙

### 접근 권한 관리

| 서비스 | 대표 | 개발팀 | 신규 멤버 |
|--------|------|--------|----------|
| GitHub (코드) | Admin | Write | Read (첫 2주) |
| 클라우드 | Admin | Developer | 없음 (요청 시) |
| 재무 데이터 | Full | 없음 | 없음 |
| 고객 데이터 | Full | 필요 시 | 없음 |
| 슬랙 | Admin | Member | Member |

## 데이터 보안

### 데이터 분류

| 등급 | 정의 | 예시 | 관리 수준 |
|------|------|------|----------|
| 기밀 | 유출 시 회사에 심각한 피해 | 소스코드, 특허 자료, 고객 데이터 | 암호화 필수, 접근 제한 |
| 내부용 | 내부에서만 사용 | 재무 데이터, 전략 문서 | 외부 공유 금지 |
| 일반 | 공개 가능 | 회사 소개, 기술 블로그 | 자유 공유 |

### 코드 보안
- 소스코드는 **Private 레포지토리**에만 저장
- API 키, 비밀번호를 코드에 직접 입력 금지 → **환경변수** 사용
- `.env` 파일은 `.gitignore`에 반드시 포함
- 커밋 전 `git diff`로 민감 정보 유출 확인

### 데이터 백업
| 대상 | 주기 | 방법 | 보관 기간 |
|------|------|------|----------|
| 소스코드 | 실시간 | GitHub (자동) | 영구 |
| 데이터베이스 | 매일 | 자동 백업 | 30일 |
| 중요 문서 | 매주 | 클라우드 동기화 | 1년 |
| 측정 데이터 | 프로젝트별 | 별도 스토리지 | 5년 |

## 장비 보안

### 노트북/PC
- 화면 잠금 (5분 이상 자리 비움 시 자동 잠금)
- 디스크 암호화 (FileVault/BitLocker) 활성화
- 최신 OS 업데이트 유지
- 공용 Wi-Fi에서 VPN 사용

### 외부 장소 업무
- 공공장소에서 화면 엿보기 주의 (프라이버시 필터 권장)
- 외부 USB/저장장치 연결 자제
- 공유 PC에 로그인하지 않기

## 사고 발생 시 대응

### 보안 사고 대응 절차

1. **발견 즉시**: 대표에게 보고 (전화)
2. **1시간 내**: 피해 범위 파악, 긴급 조치 (비밀번호 변경, 접근 차단)
3. **24시간 내**: 원인 분석, 영향 평가
4. **1주 내**: 재발 방지 대책 수립, 문서화

### 보안 사고 예시별 대응

| 사고 | 즉시 대응 |
|------|----------|
| 비밀번호 유출 | 즉시 변경, 관련 서비스 모두 점검 |
| 코드 유출 | GitHub 접근 로그 확인, 토큰 재발급 |
| 노트북 분실 | 원격 잠금/삭제, 클라우드 세션 종료 |
| 피싱 메일 클릭 | 네트워크 차단, 계정 점검 |
