# 브링엔지니어링 — 기술 부채 관리

**최종 수정일:** 2026-03-10
**참고:** Martin Fowler 기술 부채 분류, Google SRE Handbook

---

## 기술 부채란?

> 빠른 개발을 위해 의도적/비의도적으로 발생한 기술적 타협.
> 방치하면 개발 속도가 점점 느려지고, 버그가 증가하고, 새 기능 추가가 어려워짐.

---

## 부채 분류 (Martin Fowler 4분면)

```
              의도적                    비의도적
        ┌──────────────────────┬──────────────────────┐
신중한  │ "지금은 이렇게 하고  │ "나중에 보니 더 좋은 │
        │  나중에 리팩토링"     │  방법이 있었네"       │
        ├──────────────────────┼──────────────────────┤
무모한  │ "시간 없으니까       │ "이게 뭐가 문제인지  │
        │  그냥 대충"           │  모르겠는데"          │
        └──────────────────────┴──────────────────────┘
```

- **신중한 의도적**: 괜찮음. 기록하고 계획적으로 해소
- **신중한 비의도적**: 자연스러움. 코드 리뷰에서 발견
- **무모한 의도적**: 주의. 빠른 해소 필요
- **무모한 비의도적**: 위험. 학습/역량 개선 필요

---

## 기술 부채 관리 프로세스

### 1. 식별 — 언제?
- 코드 리뷰 시
- 격주 기술 싱크 미팅
- 버그 수정 중 발견

### 2. 기록 — 어떻게?
GitHub Issue에 `tech-debt` 라벨로 등록:

```
## 기술 부채 등록

**위치:** [파일/모듈명]
**유형:** 신중한 의도적 / 신중한 비의도적 / 무모한
**심각도:** 상/중/하
**영향:** [이 부채가 방치되면 어떤 문제가 생기는가]
**해결 방안:** [어떻게 해결할 수 있는가]
**예상 공수:** [시간/일]
```

### 3. 우선순위 — 기준

| 심각도 | 기준 | 해소 시기 |
|--------|------|----------|
| 상 | 서비스 장애, 보안 취약점, 데이터 손실 위험 | 다음 스프린트 |
| 중 | 개발 속도 저하, 유지보수 어려움 | 이번 분기 |
| 하 | 코드 가독성, 스타일 불일치 | 여유 있을 때 |

### 4. 해소 — 20% 규칙
> **매 스프린트의 20%는 기술 부채 해소에 할당** (Google 방식)

2주 스프린트 기준:
- 8일 중 **1.5~2일**을 리팩토링/부채 해소에 투자
- 주간 킥오프에서 부채 해소 항목 포함

## 코드 품질 기준

### 자동화 도구
| 도구 | 용도 | 설정 |
|------|------|------|
| Black/Ruff | Python 코드 포매팅 | 커밋 시 자동 실행 |
| mypy | 타입 체크 | CI에서 실행 |
| pytest | 테스트 | PR 시 자동 실행 |
| pre-commit | 커밋 전 검사 | 포매팅 + 린트 |

### 코드 리뷰 시 기술 부채 체크
- [ ] 하드코딩된 값이 있는가? → 설정 파일로
- [ ] 중복 코드가 있는가? → 공통 함수로
- [ ] TODO/FIXME 주석이 있는가? → Issue로 등록
- [ ] 테스트 없는 핵심 로직이 있는가? → 테스트 추가
- [ ] 문서가 코드와 불일치하는가? → 문서 업데이트
