## Git 워크플로우 (GitHub Flow 기반)

```
main (항상 배포 가능)
  └── feature/기능명 (기능 개발)
  └── fix/버그명 (버그 수정)
  └── experiment/실험명 (AI 실험)
```

### 브랜치 규칙

| 브랜치 | 용도 | 머지 조건 |
|--------|------|----------|
| `main` | 프로덕션 배포 | PR + 코드 리뷰 필수 |
| `feature/*` | 새 기능 개발 | PR → 리뷰 → main 머지 |
| `fix/*` | 버그 수정 | PR → 리뷰 → main 머지 |
| `experiment/*` | AI 모델 실험 | 실험 완료 후 결과 기록 → 머지 또는 삭제 |

### 커밋 메시지 규칙

```
<type>: <짧은 설명>

<상세 설명 (선택)>

타입:
- feat: 새 기능
- fix: 버그 수정
- refactor: 리팩토링
- docs: 문서
- test: 테스트
- chore: 기타 (빌드, CI 등)
```

**예시:**
```
feat: VisiScan 조도 측정 API 엔드포인트 추가
fix: Shadow AI 모델 추론 시 메모리 누수 해결
docs: API 문서 업데이트
```

## 코드 리뷰 가이드

### 리뷰어 배정
- 2인 팀이므로 **상호 리뷰** (노광민 ↔ 배정환)
- 자기 PR은 반드시 상대방이 리뷰

### 리뷰 체크리스트
- [ ] 코드가 의도한 기능을 수행하는가?
- [ ] 버그나 엣지 케이스는 없는가?
- [ ] 코드 가독성이 좋은가?
- [ ] 테스트가 포함되어 있는가?
- [ ] 보안 취약점은 없는가?
- [ ] 성능 이슈는 없는가?

### 리뷰 응답 시간
- **일반 PR**: 24시간 이내 리뷰 시작
- **긴급 PR**: 4시간 이내
- **핫픽스**: 즉시 (구두 합의 후 머지 가능)

### 리뷰 에티켓
- 코드에 대한 의견이지, 사람에 대한 평가가 아님
- "왜 이렇게 했어?" ❌ → "이 부분은 이렇게 하면 어떨까요?" ✅
- LGTM (Looks Good To Me) = 승인

## 스프린트 사이클 (2주)

```
Day 1 (월): 스프린트 플래닝 (주간 킥오프와 병행)
Day 2~9: 개발
Day 10 (금): 스프린트 리뷰/데모 (주간 리뷰와 병행)
```

### 이슈 관리 (GitHub Issues)

| 라벨 | 용도 |
|------|------|
| `priority:high` | 이번 스프린트 필수 |
| `priority:medium` | 이번 스프린트 목표 |
| `priority:low` | 다음 스프린트 |
| `bug` | 버그 |
| `feature` | 새 기능 |
| `tech-debt` | 기술 부채 |

## 배포 프로세스

### 개발 환경
```
로컬 개발 → PR → 코드 리뷰 → main 머지 → 스테이징 자동 배포
```

### 프로덕션 배포
```
main에서 태그 생성 (v1.0.0) → CI/CD 파이프라인 → 프로덕션 배포
```

### 배포 체크리스트
- [ ] 모든 테스트 통과
- [ ] 코드 리뷰 완료
- [ ] 스테이징에서 수동 테스트
- [ ] 배포 후 모니터링 (30분)
- [ ] 문제 시 즉시 롤백

## 테스트 기준

| 종류 | 기준 | 도구 |
|------|------|------|
| 단위 테스트 | 핵심 로직 커버리지 70%+ | pytest |
| 통합 테스트 | API 엔드포인트 전수 | pytest + httpx |
| AI 모델 테스트 | 정확도/성능 기준 충족 | 자체 벤치마크 |
| 수동 테스트 | 배포 전 필수 시나리오 | 체크리스트 |
