# 브링엔지니어링 — 회사 운영체계 종합 기획안

**작성일:** 2026-03-09
**작성자:** 경영전략 AI 어드바이저
**대상:** 대표 브링
**목적:** 대표 부재 시에도 자율 운영 가능한 체계 구축

---

## 1. 배경

브링엔지니어링은 현재 초기 스타트업 단계에서 조직화 단계로의 전환점에 위치. 현재 6개 파일(A_운영인사, B_사업전략, C_기술재무, F_용어사전, 개인가이드북 2명분)을 통해 체계 수립을 시도 중이나, **"왜 이것을 하는가"를 연결하는 상위 프레임워크**와 **일상 운영 리듬**이 부재한 상태.

**현재 상황 진단:**
- 4개 제품·프로젝트 동시 추진 (VisiScan, Shadow AI, Lux-Guard, FDInnovation)
- TRL 3~5 단계, 첫 유료계약 미체결 상태
- 팀원 최소 3명 이상 (노광민, 배정환, 김현서 + 대표)
- 정부지원사업 6건 수행/준비 경험
- 수상 9건, 특허 2건, MOU 3건 등 초기 레퍼런스 축적 중

## 3. 제안 — 브링엔지니어링 운영체계 마스터 프레임워크

### 3-1. 체계의 구조: 대(大) → 중(中) → 소(小)

```
[대분류] 회사 운영의 5대 축
  ├─ [중분류] 각 축별 관리 영역
  │    └─ [소분류] 실행 문서·도구·루틴
```

### 3-2. 운영 리듬 — "이 루틴만 돌면 회사가 돌아간다"

이것이 가장 핵심. **대표가 없어도 돌아가는 회사**의 핵심은 "누가 언제 무엇을 체크하는가"의 자동화.

| 주기 | 활동 | 참여자 | 산출물 | 소요시간 |
|------|------|--------|--------|----------|
| **매일** | 스탠드업 (슬랙/노션) | 전원 | 오늘 할 일 3줄 | 5분 |
| **매주 월요일** | 주간 킥오프 | 전원 | 주간 목표 확정 | 30분 |
| **매주 금요일** | 주간 리뷰·보고 | 전원 → 대표 | 주간보고서 | 30분 |
| **격주** | 기술 싱크 | 기술팀 | 기술 이슈 로그 | 1시간 |
| **매월 첫째주** | 월간 경영 리뷰 | 대표 + 핵심 | 경영 대시보드 | 1시간 |
| **매월** | 1:1 체크인 | 대표↔팀원 | 체크인 기록 | 30분/명 |
| **분기** | OKR 리뷰 & 재설정 | 전원 | OKR 점수표 | 반나절 |
| **반기** | 성과 평가 | 대표 | 개인 평가서 | 1일 |
| **연간** | 전략 워크숍 | 전원 | 연간 사업계획 | 1~2일 |

### 3-3. 의사결정 프레임워크 — "누가 결정하나"

| 결정 수준 | 기준 | 결정자 | 예시 |
|-----------|------|--------|------|
| Level 1: 자율 | 본인 업무 범위 내, 비용 0원 | 담당자 | 코드 구조 변경, 문서 작성 |
| Level 2: 공유 | 본인 업무 범위 내, 다른 팀원 영향 | 담당자 → 관련자 공유 | API 변경, 일정 조정 |
| Level 3: 합의 | 2명 이상 영향, 비용 50만 원 미만 | 관련자 합의 → 대표 보고 | 외주 발주, 장비 구매 |
| Level 4: 대표 승인 | 비용 50만 원 이상, 대외 계약 | 대표 최종 | 채용, 계약, 투자 |

### 3-4. 동기부여 & 자기주도 성장 시스템

**현재 가진 것:** 개인가이드북 (역할확장전략, 성장로드맵)
**부족한 것:** 이것이 "왜 나에게 이득인가"를 연결하는 보상 체계

제안하는 구조:

```
[개인 OKR] ←연동→ [회사 OKR]
     ↓                    ↓
[분기 성과 리뷰]    [회사 성장]
     ↓                    ↓
[역량 등급 인정]    [보상 풀 확대]
     ↓                    ↓
[인센티브·역할확장]  [팀원 보상]
```

구체적으로:
1. **스킬 레벨 시스템**: 각 모듈별 초급→중급→고급→마스터, 등급 인정 기준 명확화
2. **분기 성과 보너스**: 개인 OKR 달성률 기반 (달성률 70% 이상이면 지급)
3. **역할 확장 트리거**: 특정 스킬 레벨 달성 시 자동으로 새 역할 부여
4. **학습 시간 보장**: 주 4시간을 자기개발 시간으로 공식 보장
5. **기술 블로그/발표**: 월 1회 내부 기술 공유, 외부 발표 기회 제공

## 4. 실행 로드맵 — 우선순위 기반 12주 플랜

### Phase 1: 즉시 (1~2주) — "생존 인프라"
- [ ] 운영 리듬 캘린더 확정 (매일 스탠드업, 주간 킥오프/리뷰)
- [ ] 커뮤니케이션 프로토콜 1장 작성 (보고 채널·형식·긴급도)
- [ ] 의사결정 권한표 확정 (Level 1~4)
- [ ] 캐시플로우 12개월 예측 모델 작성

### Phase 2: 2~4주 — "사람 체계"
- [ ] 미션·비전·행동원칙 선언문 작성
- [ ] 온보딩 30-60-90일 프로그램 설계
- [ ] 성과평가 기준표 작성 (OKR 연동)
- [ ] 보상·인센티브 기준 초안

### Phase 3: 5~8주 — "품질·지식 체계"
- [ ] 납품물 QMS 체크리스트 (VisiScan 우선)
- [ ] IP 관리 대장 (특허·노하우·영업비밀 분류)
- [ ] 지식관리 시스템 구조 설계 (노션/위키)
- [ ] 퇴직·인수인계 프로토콜

### Phase 4: 9~12주 — "성장·확장 체계"
- [ ] 수익 모델 상세화 (제품별 매출 프로젝션)
- [ ] 세무·회계 관리 프로세스
- [ ] 위기 대응 매뉴얼 (인력 이탈, 사업 실패, 자금 고갈 시나리오)
- [ ] 전체 체계 통합 대시보드 구축

## 6. 제약 사항 및 해결 방안

### 제약 1: 소규모 팀으로 체계 운영 과부하 위험
- **문제:** 3~4명이 대기업 수준 프로세스를 따르면 실행보다 관리에 시간 소모
- **해결:** 각 체계를 "최소 실행 단위"로 설계. 폼 1장, 체크리스트 10항목 이내 원칙. 성장에 따라 점진 확대

### 제약 2: 대표의 사업·조직 경험 부족
- **문제:** 체계를 만들어도 운영 노하우 부족 시 형식적으로 흐를 위험
- **해결:** 운영 리듬(주간 킥오프, 월간 리뷰)을 "습관"으로 정착시키는 것이 최우선. 12주 동안 1개씩 추가하되, 한 번에 전부 도입하지 않음

### 제약 3: 매출 미발생 상태에서 보상 체계 설계 한계
- **문제:** 현금이 없는데 인센티브를 약속하면 신뢰 상실 위험
- **해결:** 금전적 보상 전에 "역할 확장"과 "공식 인정"을 먼저 설계. 스톡옵션/성과급은 첫 매출 발생 이후 단계적 도입

### 제약 4: 4개 프로젝트 동시 추진으로 집중도 분산
- **문제:** 자원 한정 상태에서 모든 프로젝트를 동등하게 추진하면 전부 느려짐
- **해결:** VisiScan을 "1순위 수익원"으로 확정하고, 나머지는 마일스톤 기반 자원 배분. 분기 OKR에서 우선순위 명시적으로 반영

### 제약 5: 문서·체계가 실제 사용되지 않을 위험
- **문제:** 엑셀 파일을 만들어도 아무도 안 열어보면 의미 없음
- **해결:** 노션 등 협업 도구에 "살아있는 문서"로 이관. 주간 리뷰에서 실제 문서를 열어 함께 업데이트하는 루틴 강제

## 7. 최종 정리 — 지금 바로 의사결정이 필요한 3가지

**첫째,** 파일 카테고리 재편: 현재 A-B-C-F → A~G 7개 체계로 확장할 것인지
**둘째,** 운영 도구 확정: 엑셀 유지 vs 노션/슬랙 기반 전환
**셋째,** 12주 실행 로드맵 시작 시점 및 첫 주 착수 항목 확정

---

> **핵심 한줄:** 체계란 "규칙을 만드는 것"이 아니라 **"규칙이 습관이 되는 리듬을 설계하는 것"**. 지금 가장 시급한 건 문서 10개를 더 만드는 것이 아니라, **주간 킥오프 30분을 4주 연속 실행하는 것**.
